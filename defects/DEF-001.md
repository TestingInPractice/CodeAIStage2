# Defect: Password stored with SHA-256 instead of bcrypt

## Description
The `_hash_password` function in `app/main.py` uses `hashlib.sha256` to hash passwords. The requirements explicitly specify bcrypt for password hashing because it provides automatic salting and is a well-tested, slow hashing algorithm suitable for password storage. SHA-256 is a fast hash without salting, making it vulnerable to rainbow table and brute-force attacks.

## Steps to Reproduce
1. Register a user via `POST /api/register` with a valid payload.
2. Read `users.json` and inspect the `password` field.
3. Observe that the stored value is a 64-character hex string (SHA-256), not a `$2b$` prefixed bcrypt hash.

## Expected Behavior
The password field in `users.json` should contain a bcrypt hash (starting with `$2b$` or `$2a$`, 60 characters long).

## Actual Behavior
The password field contains a plain SHA-256 hex digest (64 characters), which is not salted and is trivially reversible via rainbow tables.

## Suggested Fix
Replace the `_hash_password` function with a bcrypt-based implementation:

```python
import bcrypt

def _hash_password(password: str) -> str:
    return bcrypt.hashpw(password.encode("utf-8"), bcrypt.gensalt()).decode("utf-8")
```

Add `bcrypt` to `requirements.txt` if not already present.
